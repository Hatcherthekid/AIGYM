## 1. 背景与目标

### 1.1 问题
用户有3年健身经验，当前使用：
- 训记APP记录训练（截图形式）
- 微信/备忘录询问AI训练建议
- 手动调整计划

痛点：
1. 训练记录散落在APP截图中，无法追溯趋势
2. AI建议无上下文（不知道上周练了什么、当前伤病状态）
3. 计划调整依赖对话，无结构化沉淀

### 1.2 目标
构建一个**个人健身数据中枢**，实现：
1. 训练记录可结构化存储 + 可视化展示
2. AI基于完整上下文给出个性化建议
3. 计划调整可双向编辑（AI生成 / 手动修改）

### 1.3 非目标（明确不做）
| 不做 | 原因 |
|-----|------|
| 饮食热量计算 | 超出训练核心场景，已有薄荷等工具 |
| 睡眠追踪 | 需额外硬件/授权，ROI低 |
| 社交/打卡 | 个人使用，无社交需求 |
| 视频动作指导 | 内容成本高，非核心需求 |
| 多用户支持 | MVP阶段仅服务单一用户 |

## 2. 目标用户与核心场景

### 2.1 用户画像
- **身份**：有3年+训练经验的健身爱好者
- **特征**：有肩/踝伤病史，需康复指导；偏好力量训练，偶尔备赛HYROX
- **现状**：训练认知成熟，需要"数据化+智能化"辅助决策

### 2.2 核心场景（5个）

| 场景ID | 场景名 | 触发条件 | 核心价值 |
|-------|-------|---------|---------|
| S1 | 记录训练 | 训练后发送截图/文字 | 数据沉淀 |
| S2 | 今天练什么 | 主动询问/定时推送 | 决策辅助 |
| S3 | 动作替代 | 器械不可用/伤病限制 | 灵活适配 |
| S4 | 康复指导 | 疼痛反馈 | 风险控制 |
| S5 | 周复盘 | 周日自动生成 | 长期规划 |

## 4. 方案概要与关键流程

### 4.1 系统架构

**目录结构（单仓库管理）：**
```
fitness-assistant/
├── backend/                 # FastAPI后端服务
│   ├── api/                 # REST API端点
│   ├── models/              # SQLAlchemy模型
│   ├── services/            # 业务逻辑
│   │   ├── ai/              # AI角色（训练师/康复师）
│   │   ├── parser.py        # 文本解析
│   │   ├── ocr.py           # OCR识别
│   │   └── feishu.py        # 飞书集成
│   └── main.py              # 服务入口
├── scripts/                 # 数据库初始化等脚本
├── docs/                    # 项目文档
└── docker-compose.yml       # 部署配置
```

**数据流：**
```
用户输入层              AI处理层                数据层                 展示层
├─ 文本输入      ───→   意图识别    ───→     PostgreSQL    ───→    飞书多维表格
├─ 截图(OCR)     ───→   信息提取    ───→     Redis缓存     ───→    飞书日历
└─ 混合输入      ───→   角色路由    ───→     离线队列      ───→    飞书Dashboard
                        ↓
                    训练师/康复师（同一代码库）
                        ↓
                    结构化输出
```

### 4.2 关键流程

#### 流程1：记录训练 - 逐组对话模式（推荐）

```
用户: /记录
    ↓
机器人: 📋 开始记录推日训练，动作1是什么？
    ↓
用户: 卧推
    ↓
机器人: 🏋️ 卧推 - 第1组，重量？（如：60kg热身）
    ↓
用户: 60kg
    ↓
机器人: 几次？RPE？（如：12次RPE6）
    ↓
用户: 12次RPE6
    ↓
机器人: ✅ 已记录：卧推 60kg×12次 (热身组, RPE6)
        下一组？（输重量继续，或/下一动作，或/完成）
    ↓
用户: 80kg
    ↓
...继续直到/完成
    ↓
写入飞书表格（逐组，一行=一组）
    ↓
回复总结：今日训练完成！总容量5200kg
```

**快捷模式（简单动作）：**
```
User: /记录 深蹲 100kg 5次×5组 正式
AI: ✅ 已记录深蹲 5组，总容量2500kg
```

**图片识别模式：**
```
User: /llm [训记截图]
AI: 🖼️ 识别到训练记录：
    卧推：60×12(热身) / 80×8(热身) / 100×5(正式) / 100×5(正式) / 90×6(递减)
    [确认记录] [修改] [跳过]
```

#### 流程2：今天练什么

```
用户询问：今天练什么？
    ↓
AI读取用户状态快照：
  - 最近3天训练内容
  - 当前酸痛分布
  - 时间约束
    ↓
角色判断 → 训练师
    ↓
生成训练单（部位+动作+参数）
    ↓
回复建议 + 可选同步到飞书日历
```

**示例：**
```
User: 今天练什么？只有45分钟
AI: 分析：昨天推，前天休息，背已恢复48小时
    
    建议：拉训练
    1. 引体向上 4组×力竭
    2. 杠铃划船 3组×8 @ 50kg
    3. 高位下拉 3组×12
    
    预估：40-45分钟
    
    [添加到日历]
```

#### 流程3：修改训练记录（只能机器人修改）

```
用户: /修改 今天 卧推 第3组 重量为90kg
    ↓
后端查询飞书表格，找到对应记录
    ↓
确认修改内容：100kg×5次 → 90kg×5次，容量500→450
    ↓
发送确认卡片给用户
    ↓
用户确认
    ↓
调用飞书API更新表格
    ↓
回复：✅ 已修改，飞书表格已更新
```

**删除记录：**
```
User: /删除 今天 卧推 第5组
AI: 🔍 找到记录：卧推 第5组 90kg×6次
    确认删除？
    [确认] [取消]

User: /删除 昨天 所有记录
AI: ⚠️ 将删除昨天的8组记录，确认？
    [确认删除] [取消]
```
    ↓
发送飞书消息卡片：
    "检测到冲突：
     飞书版本：重量60kg（修改于10:23）
     本地版本：重量65kg（修改于10:25）
     [使用飞书] [使用本地] [保留两者]"
```

### 4.3 数据模型（关键更新）

#### 飞书表格设计（一行 = 一组）

**主表：训练记录**

| 字段 | 类型 | 说明 | 示例 |
|-----|------|------|------|
| 日期 | Date | 训练日期 | 2026-03-14 |
| 训练日ID | Text | 唯一标识 | 20260314-PUSH |
| 序号 | Number | 动作序号 | 1, 2, 3... |
| 动作 | Text | 动作名称 | 卧推 |
| 组号 | Number | 第几组 | 1, 2, 3... |
| 重量 | Text | 含单位 | 100kg |
| 次数 | Number | 完成次数 | 5 |
| RPE | Number | 自觉用力程度 1-10 | 9 |
| 组类型 | Select | 热身/正式/递减/超级组/力竭 | 正式 |
| 备注 | Text | 该组感受/问题 | 左肩不适 |
| 容量 | Formula | 自动计算 | 重量×次数 |

**汇总表（公式自动生成）**
- 按动作分组统计
- 按日期统计总容量
- 周/月趋势图表

#### 与旧模型对比

| 场景 | 旧模型（一行=动作） | 新模型（一行=组） |
|-----|-------------------|------------------|
| 卧推5组不同重量 | 一行，sets字段存JSON数组 | 5行，每组独立记录 |
| 查看第3组RPE | 解析JSON | 直接读取RPE列 |
| 修改第3组重量 | 改JSON数组 | 改对应行的重量列 |
| 统计递增组容量 | 复杂计算 | 按组类型筛选求和 |

完整Schema见：`docs/fitness-system-architecture.md`

## 5. 验收标准

### 5.1 功能验收

| 功能 | 验收标准 | 验证方式 |
|-----|---------|---------|
| 逐组记录 | `/记录` 支持逐组对话录入，每组包含重量/次数/RPE/组类型/备注 | 完整记录一次训练（3动作×5组），验证飞书表格每组一行 |
| 快捷记录 | `/记录 动作 重量 次数×组数` 一句话录入多组 | 测试10种常见格式，全部正确解析 |
| 图片记录 | `/llm [图片]` 识别训记截图，提取逐组数据 | 人工抽查10张截图，逐组准确率≥90% |
| 修改记录 | `/修改 日期 动作 组号 字段=新值` 正确更新 | 修改重量/次数/RPE/备注，验证飞书表格更新 |
| 删除记录 | `/删除` 支持删除单组或整天记录 | 删除后验证飞书表格行数减少 |
| 今天练什么 | 询问后30秒内返回个性化训练单（含逐组参数） | 验证建议包含：动作、每组重量次数、组类型 |
| 飞书写入 | 后端调用飞书API，写入后5秒内表格可见 | 检查飞书表格记录存在性和字段完整性 |
| 表格只读 | 普通用户无法直接编辑表格，只能通过指令修改 | 尝试用非机器人账号编辑表格，验证被拒绝 |

### 5.2 数据准确性

| 指标 | 标准 | 验证方式 |
|-----|------|---------|
| 容量计算 | 重量×次数×组数，误差=0 | 单元测试覆盖 |
| 1RM估算 | Epley公式：1RM = 重量 × (1 + 次数/30) | 对比已知PR验证 |
| 部位分类 | 动作正确归类到胸/背/腿/肩/手臂/核心 | 抽查验证 |

### 5.3 性能标准

| 场景 | 标准 |
|-----|------|
| API响应 | P95 < 500ms |
| 飞书同步延迟 | < 10秒 |
| OCR处理时间 | < 5秒 |
