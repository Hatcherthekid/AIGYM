## 1. 背景与目标

### 1.1 问题
用户有3年健身经验，当前使用：
- 训记APP记录训练（截图形式）
- 微信/备忘录询问AI训练建议
- 手动调整计划

痛点：
1. 训练记录散落在APP截图中，无法追溯趋势
2. AI建议无上下文（不知道上周练了什么、当前伤病状态）
3. 计划调整依赖对话，无结构化沉淀

### 1.2 目标
构建一个**个人健身数据中枢**，实现：
1. 训练记录可结构化存储 + 可视化展示
2. AI基于完整上下文给出个性化建议
3. 计划调整可双向编辑（AI生成 / 手动修改）

### 1.3 非目标（明确不做）
| 不做 | 原因 |
|-----|------|
| 饮食热量计算 | 超出训练核心场景，已有薄荷等工具 |
| 睡眠追踪 | 需额外硬件/授权，ROI低 |
| 社交/打卡 | 个人使用，无社交需求 |
| 视频动作指导 | 内容成本高，非核心需求 |
| 多用户支持 | MVP阶段仅服务单一用户 |

## 2. 目标用户与核心场景

### 2.1 用户画像
- **身份**：有3年+训练经验的健身爱好者
- **特征**：有肩/踝伤病史，需康复指导；偏好力量训练，偶尔备赛HYROX
- **现状**：训练认知成熟，需要"数据化+智能化"辅助决策

### 2.2 核心场景（5个）

| 场景ID | 场景名 | 触发条件 | 核心价值 |
|-------|-------|---------|---------|
| S1 | 记录训练 | 训练后发送截图/文字 | 数据沉淀 |
| S2 | 今天练什么 | 主动询问/定时推送 | 决策辅助 |
| S3 | 动作替代 | 器械不可用/伤病限制 | 灵活适配 |
| S4 | 康复指导 | 疼痛反馈 | 风险控制 |
| S5 | 周复盘 | 周日自动生成 | 长期规划 |

## 4. 方案概要与关键流程

### 4.1 系统架构

**目录结构（单仓库管理）：**
```
fitness-assistant/
├── backend/                 # FastAPI后端服务
│   ├── api/                 # REST API端点
│   ├── models/              # SQLAlchemy模型
│   ├── services/            # 业务逻辑
│   │   ├── ai/              # AI角色（训练师/康复师）
│   │   ├── parser.py        # 文本解析
│   │   ├── ocr.py           # OCR识别
│   │   └── feishu.py        # 飞书集成
│   └── main.py              # 服务入口
├── scripts/                 # 数据库初始化等脚本
├── docs/                    # 项目文档
└── docker-compose.yml       # 部署配置
```

**数据流：**
```
用户输入层              AI处理层                数据层                 展示层
├─ 文本输入      ───→   意图识别    ───→     PostgreSQL    ───→    飞书多维表格
├─ 截图(OCR)     ───→   信息提取    ───→     Redis缓存     ───→    飞书日历
└─ 混合输入      ───→   角色路由    ───→     离线队列      ───→    飞书Dashboard
                        ↓
                    训练师/康复师（同一代码库）
                        ↓
                    结构化输出
```

### 4.2 关键流程

#### 流程1：记录训练（主流程）

```
用户发送训记截图
    ↓
AI OCR提取：动作/组数/重量/次数
    ↓
结构化存储 → PostgreSQL
    ↓
同步到飞书多维表格
    ↓
更新用户状态快照
    ↓
回复确认消息 + 简短反馈
```

**示例：**
```
User: [发送训记截图]
AI: 识别到训练记录：
    卧推 5组×(60×10, 70×8, 75×6, 70×7, 65×8)
    上斜哑铃卧推 3组×12 @ 22kg
    总容量：12,800kg | 胸部
    
    已记录，同步到飞书表格。
```

#### 流程2：今天练什么

```
用户询问：今天练什么？
    ↓
AI读取用户状态快照：
  - 最近3天训练内容
  - 当前酸痛分布
  - 时间约束
    ↓
角色判断 → 训练师
    ↓
生成训练单（部位+动作+参数）
    ↓
回复建议 + 可选同步到飞书日历
```

**示例：**
```
User: 今天练什么？只有45分钟
AI: 分析：昨天推，前天休息，背已恢复48小时
    
    建议：拉训练
    1. 引体向上 4组×力竭
    2. 杠铃划船 3组×8 @ 50kg
    3. 高位下拉 3组×12
    
    预估：40-45分钟
    
    [添加到日历]
```

#### 流程3：飞书表格编辑回写

```
用户在飞书表格修改重量：60kg → 65kg
    ↓
飞书Webhook触发 → 后端API
    ↓
验证版本号（乐观锁）
    ↓
更新PostgreSQL
    ↓
如影响计划，触发重新计算
    ↓
同步完成
```

**冲突处理：**
```
如果后端也有未同步的修改：
    ↓
发送飞书消息卡片：
    "检测到冲突：
     飞书版本：重量60kg（修改于10:23）
     本地版本：重量65kg（修改于10:25）
     [使用飞书] [使用本地] [保留两者]"
```

## 5. 验收标准

### 5.1 功能验收

| 功能 | 验收标准 | 验证方式 |
|-----|---------|---------|
| 文本记录 | 输入"卧推4组，60kg×10"，正确解析并存入数据库 | API测试：POST /api/v1/logs，验证response |
| OCR记录 | 上传训记截图，正确提取动作/组数/重量 | 人工抽查5张截图，准确率≥90% |
| 今天练什么 | 询问后30秒内返回个性化训练单 | 端到端测试，验证内容含部位+动作+预估时间 |
| 飞书同步 | 后端写入后10秒内飞书表格可见 | 检查飞书表格更新时间戳 |
| 表格回写 | 飞书编辑后，后端数据库30秒内更新 | Webhook日志 + 数据库查询验证 |
| 离线恢复 | 断网期间编辑，恢复后自动同步 | 模拟断网，验证队列+同步流程 |

### 5.2 数据准确性

| 指标 | 标准 | 验证方式 |
|-----|------|---------|
| 容量计算 | 重量×次数×组数，误差=0 | 单元测试覆盖 |
| 1RM估算 | Epley公式：1RM = 重量 × (1 + 次数/30) | 对比已知PR验证 |
| 部位分类 | 动作正确归类到胸/背/腿/肩/手臂/核心 | 抽查验证 |

### 5.3 性能标准

| 场景 | 标准 |
|-----|------|
| API响应 | P95 < 500ms |
| 飞书同步延迟 | < 10秒 |
| OCR处理时间 | < 5秒 |
