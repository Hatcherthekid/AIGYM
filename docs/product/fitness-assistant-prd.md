> 状态：active

> 更新：2026-03-22  
> 设计真相：仓库文档优先，飞书/OpenClaw 后台配置需回填到仓库

---

## 1. 背景与目标

当前训练相关体验分散在几个地方：

- 训记类 App 负责高频记录
- 飞书对话负责和 AI 讨论今天练什么、伤病、训练调整
- 训练截图、设备屏幕、零散数字无法稳定沉淀成可复用的结构化历史

当前飞书 MVP 的目标不是做一个完整记录器，也不是先做正式 App，而是先把“训练记录 + 教练调整”的核心链路打通，验证这套训练 agent 是否真的更聪明、更有上下文。

这期的目标有三个：

1. 让训练数据能结构化沉淀，成为 AI 调整动作和计划的依据  
2. 让 agent 在训练前、训练中、训练后都能给出可执行判断  
3. 为未来独立记录器 App 预埋稳定内核，而不是把产品真相锁死在飞书表格里

### 非目标

- 不做正式 App UI
- 不做营养/热量体系
- 不做 HealthKit / Apple Watch
- 不做训记级记录交互
- 不做多用户/教练端/社区

---

## 2. 产品定位

### 当前阶段

`飞书 = 运行壳，OpenClaw = 对话与路由层，飞书多维表 = 当前运行时结构化真相层。`

当前阶段允许用飞书来模拟数据库和管理台，但不允许把聊天历史或 prompt 当成长期状态来源。

### 长期方向

长期产品不是飞书机器人，而是“训练记录器 + 教练 agent”的组合：

- 记录器负责高频训练录入
- 教练 agent 负责计划、调整、解释、复盘

未来独立记录器 App 会与飞书并行一段时间，因此当前训练内核必须可迁移。

---

## 3. 目标用户与核心场景

### 用户画像

- 有多年训练经验
- 需要在伤病/疲劳/时间限制下持续训练
- 已经在用训练记录工具，但希望 AI 能基于真实历史给出更智能的调整

### 核心场景

1. `开始训练`
   - 今天练什么
   - 只有 40 分钟
   - 肩和踝康复优先

2. `记录一组`
   - 文本记录重量、次数、RPE
   - 训练截图/设备屏幕作为补充录入

3. `批量导入训练`
   - 一次发一堆动作
   - 发训记截图后逐动作确认

4. `训练中调整`
   - 这一组感觉不对
   - 器械没了
   - 肩不舒服
   - 只剩 20 分钟

5. `下一组建议`
   - 要不要加重量
   - 该不该降级
   - 节奏、动作幅度、停顿怎么改

6. `结束训练`
   - 今天到这停不停
   - 做什么收尾
   - 下次怎么改

7. `训练周边决策`
   - 很饿但想练，先吃还是先练
   - 4x4 / Zone2 等协议怎么调

## 4. Scope / Out of Scope

### Scope

- 训练 session 状态机
- set 级结构化记录
- 训练截图/设备屏幕的候选导入
- 批量训练导入候选与确认
- 动作标准化与替代动作
- 下一组建议
- 结束训练短复盘
- 训练前补给/是否开练
- 4x4 / Zone2 等有氧协议调参
- 飞书表格、日历、文档作为当前运行时展示层

### Out of Scope

- 营养管理体系
- 缺口计算与长期减脂规划
- 睡眠与体重系统
- 视频动作识别
- 完整后端迁移

## 5. 核心流程

### 5.1 开始训练

用户输入：

- 今天练什么
- 我只有 40 分钟
- 肩和踝康复优先

系统行为：

- 查询最近训练、当前限制条件、active session
- 生成今日训练草案
- 如无 active session，则创建一条 session

输出形式：

- 一句话结论
- 3 到 5 条训练安排
- 必要时同步到飞书日历

### 5.2 记录一组

用户输入：

- 下拉第一组 26kg×10，手臂累，背没感觉
- 卧推 80 8
- 训记截图 / PM5 屏幕图

系统行为：

- 解析结构化 set
- 查询当前 active session 和动作标准库
- 文本输入可直接记；图片输入先生成候选，确认后写入

输出形式：

- 确认记录结果
- 必要时带一条动作/节奏提示

### 5.3 批量导入训练

用户输入：

- 我一次发一堆动作，你帮我导入
- 这张训记截图帮我整理

系统行为：

- 解析为候选 workout payload
- 按动作或整批进入确认态
- 确认后才写入正式 `Set`

输出形式：

- 候选动作/组数列表
- 可确认、跳过、整批导入
### 5.4 修改或删除一组

用户输入：

- 第三组改成 75kg
- 删除最后一组
- 那组其实是热身

系统行为：

- 定位目标 set
- 修改或删除
- 返回变更确认

### 5.5 换动作 / 下一组建议

用户输入：

- 推还是不舒服，今天别练推了
- 第二组还是没感觉，想加重量试试
- 引体 8/6/5/4/3，下一组咋办

系统行为：

- 查询当前 session、历史表现、约束条件
- 优先给出更安全、可执行的调整
- 不直接盲目加重

### 5.6 结束训练

用户输入：

- 今天做完了
- 腰有点酸但不是刺痛

系统行为：

- 结束 active session
- 生成短复盘
- 给出收尾动作和下次调整提示

## 6. 数据与模型原则

虽然当前运行在飞书里，但数据模型必须按未来 App 内核来设计，而不是按飞书字段的临时便利来设计。

### 核心对象

- `Session`
- `Set`
- `Exercise`
- `Constraint`
- `SessionSummary`
- `SourceArtifact`

### 核心规则

- 同时只能有一个 `active session`
- 每一组训练必须是独立对象
- 图片/截图先进入候选导入，不直接成为正式记录
- 所有建议都必须引用结构化历史，不允许从聊天历史脑补

### 提交与投影规则

- 用户层 skill 只负责生成结构化结果和决定要写哪些主真相对象
- 系统内置提交流程固定执行：
  1. 提交主真相对象
  2. 同步关键投影
  3. 投影失败则标记 `pending_sync`
- 主真相层：
  - `sessions`
  - `sets`
  - `exercise_catalog`
  - `constraints`
- 投影层：
  - `session_summaries`
  - 飞书日历
  - 飞书文档 / 展示视图

## 7. Agent 设计原则

当前不采用人格型角色系统。Agent 采用：

- 一个薄 `router`
- 多个任务型 `skills`

当前正式技能集：

- `start_session`
- `log_set`
- `bulk_ingest_workout`
- `amend_set`
- `swap_exercise`
- `suggest_next_set`
- `end_session`
- `query_context`
- `pre_workout_decision`
- `conditioning_protocol_adjust`

图片识别必须遵守：

- 未确认不得入库
- 动作名、重量、次数异常时进入确认态
- 禁止模型根据模糊图像补齐不存在的数据

## 8. 验收标准

### 8.1 记录可靠

- 文本记录一组能稳定落到单组对象
- 修改/删除能命中明确 set
- 图片导入不会直接幻觉写入正式记录

### 8.2 决策聪明

- “今天练什么”能基于历史和约束给出合理方案
- “还是没感觉”优先调整执行质量，而不是直接加重
- “今天不练推”能给替代动作和回归门槛

### 8.3 文档一致

- PRD、架构文档、README、backlog、decisions、iteration-log 对 source of truth 的描述完全一致

---

## 9. 风险与未决问题

### 主要风险

1. 图片导入幻觉严重，污染结构化历史
2. 飞书后台配置和仓库文档可能漂移
3. 训练周边决策容易滑向营养/泛健康范围
4. 当前飞书交互不适合模拟正式记录器体验

### 当前默认选择

- 仓库文档优先
- 飞书继续作为 MVP 运行容器
- FastAPI/SQLAlchemy 保留为未来适配层
- 未来记录器 App 一定会做，但不在当前迭代开发
