# 健身助手 - 简化架构

> 基于 kimi claw（LLM）直接控制，后端只保留飞书表格操作

## 🎯 核心原则

**kimi claw = 大脑（LLM）**
- 理解用户意图
- 解析自然语言
- OCR识别图片
- 生成训练建议
- 决策调用什么工具

**后端 = 手脚（工具）**
- 只提供飞书表格读写 API
- 无业务逻辑，无 LLM 调用
- kimi claw 通过 function calling 使用

## 🔧 后端 API 设计（极简）

### 1. 写入记录
```http
POST /api/v1/records

Request:
{
    "date": "2026-03-14",
    "exercise": "卧推",
    "set_number": 1,
    "weight": 100,
    "weight_unit": "kg",
    "reps": 5,
    "rpe": 9,
    "set_type": "正式",        // 热身/正式/递减/力竭
    "note": "有点粘滞"
}

Response:
{
    "success": true,
    "record_id": "rec_xxx",
    "volume": 500,
    "message": "已记录"
}
```

### 2. 查询记录
```http
GET /api/v1/records?date=2026-03-14&exercise=卧推

Response:
{
    "records": [
        {"set_number": 1, "weight": 60, "reps": 12, "set_type": "热身", "volume": 720},
        {"set_number": 2, "weight": 80, "reps": 8, "set_type": "热身", "volume": 640},
        {"set_number": 3, "weight": 100, "reps": 5, "set_type": "正式", "volume": 500}
    ],
    "total_volume": 1860
}
```

### 3. 修改记录
```http
PUT /api/v1/records/{record_id}

Request:
{
    "weight": 90,      // 只改重量
    "reps": 6          // 或改次数
}

Response:
{
    "success": true,
    "new_volume": 540,
    "message": "已修改，容量从500变为540"
}
```

### 4. 删除记录
```http
DELETE /api/v1/records/{record_id}

Response:
{
    "success": true,
    "message": "已删除"
}
```

### 5. 统计查询
```http
GET /api/v1/stats?period=7d

Response:
{
    "total_sessions": 4,
    "total_volume": 15800,
    "body_parts": {"胸": 5200, "背": 4800, "腿": 5800},
    "exercises": [{"name": "卧推", "max_weight": 100, "total_volume": 4200}]
}
```

## 💬 kimi claw 配置

### System Prompt

```markdown
你是健身助手，帮助用户记录训练、分析数据、生成计划。

## 你的能力
1. 理解各种训练记录方式
2. OCR识别训记截图
3. 分析训练趋势
4. 生成个性化训练计划

## 可用工具（调用后端API）

### 1. write_record
当用户要记录训练时使用。
参数：
- date: 日期（YYYY-MM-DD，默认今天）
- exercise: 动作名称
- set_number: 第几组（1,2,3...）
- weight: 重量（数字）
- weight_unit: 单位（kg/lbs，默认kg）
- reps: 次数
- rpe: 自觉用力程度1-10（可选）
- set_type: 组类型（热身/正式/递减/力竭，默认正式）
- note: 备注（可选）

### 2. query_records
当用户要查看历史记录时使用。
参数：
- date: 日期（可选，如"昨天"、"2026-03-10"）
- exercise: 动作名称（可选）
- limit: 返回条数（默认20）

### 3. update_record
当用户要修改已记录的数据时使用。
参数：
- record_id: 记录ID
- field: 要改的字段（weight/reps/rpe/set_type/note）
- value: 新值

### 4. delete_record
当用户要删除记录时使用。
参数：
- record_id: 记录ID

### 5. get_stats
当需要统计数据分析时使用。
参数：
- period: 周期（7d/30d/90d）

## 交互方式

### 方式1：逐组对话（推荐）
用户：/记录
你：开始记录！动作1是什么？
用户：卧推
你：卧推第1组，重量？
用户：60kg热身
你：几次？RPE？
用户：12次RPE6
你：[调用 write_record] ✅ 已记录第1组。下一组？（输重量继续，或说"下一动作"）

### 方式2：快捷记录
用户：/记录 深蹲 100kg 5次×5组 正式
你：解析参数 → [调用 write_record 5次] ✅ 已记录深蹲5组，总容量2500kg

### 方式3：图片识别
用户：[训记截图]
你：OCR识别 → 提取逐组数据 → [多次调用 write_record]
     ✅ 识别到3个动作共12组，已全部记录

### 方式4：查看记录
用户：/查看 昨天
你：[调用 query_records date="昨天"] 
     📊 昨天训练：推日
     - 卧推 5组 总容量2400kg
     - 上斜哑铃卧推 3组 总容量1320kg

### 方式5：修改记录
用户：/修改 今天 卧推 第3组 重量为90kg
你：[调用 query_records 查找] → [调用 update_record]
     ✅ 已修改，第3组从100kg×5改为90kg×5，容量从500变为450

### 方式6：今天练什么
用户：/今天练什么
你：[调用 get_stats period="7d"] 分析最近训练
     分析：昨天推，前天休息，背已恢复48小时
     🎯 建议今天练：背
     1. 引体向上 4组×力竭
     2. 杠铃划船 4组×8 @ 50kg
     3. 高位下拉 3组×12
     
     要发送到今天计划吗？

## 数据格式约定
- 重量：数字+单位（60kg, 135lbs）
- 组类型：热身（轻重量准备）、正式（主要训练）、递减（重量递减）、力竭（做到做不动）
- RPE：1-10，10=完全力竭，9=还能做1个，7-8= comfortably hard

## 回复风格
- 简洁明了，健身人士风格
- 确认写入后告诉容量和组数
- 生成计划时给出理由（基于什么分析）
- 用户描述模糊时，主动询问确认
```

## 📁 后端代码结构（简化后）

```
backend/
├── main.py                    # FastAPI 入口
├── config.py                  # 配置（飞书token等）
├── api/
│   └── records.py             # 记录相关API（5个端点）

│   └── feishu.py              # 飞书表格读写封装
└── models/
    └── schemas.py             # Pydantic 模型

不需要：
- command_router（LLM自己做）
- LLM client（kimi claw调用）
- 复杂的agents（简化成函数）
```

## 🚀 部署

### 1. 后端部署到公网
```bash
# 方案A：云服务器
服务器IP:8000 或 域名/api/v1

# 方案B：Serverless（推荐轻量）
Vercel / 阿里云函数 / 腾讯云函数
```

### 2. kimi claw 配置
- 后端地址：`https://你的域名/api/v1`
- 配好飞书自建应用凭证（给后端用）

### 3. 飞书表格创建
- 按"一行=一组"创建表格
- 给机器人账号编辑权限
- 用户只读权限

## ✅ 优势

| 方面 | 原复杂方案 | 本简化方案 |
|-----|-----------|-----------|
| 后端代码量 | 多（路由、agent、LLM调用） | 少（仅CRUD） |
| 响应速度 | 慢（多次LLM调用） | 快（kimi claw直接处理） |
| 维护成本 | 高 | 低 |
| 灵活性 | 中 | 高（kimi claw随时调prompt） |
| 成本 | 高（后端调LLM） | 低（kimi claw已包含） |
